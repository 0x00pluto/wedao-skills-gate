查询逻辑现在需要检查用户拥有的**所有角色**，只要其中任何一个角色包含了请求的技能，并且没有跨租户，就予以放行。

```plaintext

校验规则：
1. 接收参数：`key` (企业授权码), `email` (员工邮箱), `skillId` (目标技能)。
2. 第一层（企业全局防线）：
   - 查询 `companies` 确认 `enterprise_key` 存在且 `expiry_at` 未过期。记录其 company_id。

3. 第二层（员工状态与归属）：
   - 查询 `users` 确认 `email` 存在且 status='active'。
   - 校验归属：该用户的 `company_id` 必须与企业防线中的 `company_id` 一致。
4. 第三层（多角色聚合权限校验）：
   - 联表查询该用户的 `user_roles`。

   - 关联 `role_skills` 表，检查这些合法的 `role_id` 集合中，是否至少有一个角色包含了请求的 `skillId`。
   - (可选) 一并检查 `user_skills` 表中是否有个人特例权限。
5. 只要第三层中任意一个条件满足，即放行并签发 GitHub Token。否则返回对应的 403 错误原因。
```
